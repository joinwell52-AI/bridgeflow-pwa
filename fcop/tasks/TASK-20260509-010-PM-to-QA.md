#### §3 测试场景设计（详细列表）

按 4 个分组组织，每场景含：**输入 / 操作步骤 / 期望输出 / 通过标准 / 测试类型**（unit / integration / 手工）。

##### §3.1 PersistentStore atomic-write（决策 1）

- **TS-1.1** 正常 write → 文件等效 + JSON 解析等效
- **TS-1.2** write-temp 阶段中断（mock fs.writeFile throw）→ 原 agents.json 保持 / .tmp 文件可见用作错误诊断
- **TS-1.3** rename 阶段中断（mock fs.rename throw）→ 原 agents.json 保持 / 抛错可读
- **TS-1.4** 读不存在文件 → 返回 `[]`，不抛
- **TS-1.5** 读 corrupted JSON → 抛 `RuntimeBootstrapError`，错误消息含文件路径
- **TS-1.6** 大量并发 upsert（如 100 次）→ 最终 agents.json 状态正确（即使无并发锁，atomic-rename 保证不出现半残文件）

##### §3.2 RuntimeBootstrap reconciliation（决策 2 + 决策 3）

- **TS-2.1** records=[] → SDK.list 不调用，跳过 reconcile，输出空 report
- **TS-2.2** records=2 + SDK list 含两者 → 全部 success
- **TS-2.3** 场景 X（本地多了）→ 标 failed + report.orphaned 含该 record + agents.json 里 record.status=error
- **TS-2.4** 场景 Y（SDK 多了）→ report.foreign 含该 sdk_agent_id + agents.json **不被新加 record**
- **TS-2.5** SDK.resume 抛错 → markFailed + report.failed 含该 record + 错误原因可读
- **TS-2.6** Reconciliation 期间触发 register → throw `RuntimeNotReadyError`
- **TS-2.7** RuntimeBootstrap 幂等性：连续跑 2 次，结果应等价
- **TS-2.8** SDK.list 网络超时 → 全 record 标 failed（reason=可读）+ runtime 不阻断启动（按决策 2 末尾"单 record 失败不阻断启动"，但全 list 失败需 ADMIN 决策）—— **本场景设计上是开放的，QA 应在文档里指出这个边界条件，请 PM 与 DEV 在 Phase A 实施期间敲定**

##### §3.3 AgentRegistry 6 方法（决策 1 + 决策 3）

- **TS-3.1 register 正常**：fixture spec → record + agents.json 持久化
- **TS-3.2 register 入参不合 schema**：throw `ValidationError`，agents.json 不变
- **TS-3.3 register layer=admin**：throw `LayerViolationError`，**SDK.create 不被调用**（spy 验证）
- **TS-3.4 register SDK.create 抛错**：throw + agents.json 不被写入（atomic 保证）
- **TS-3.5 resume 找不到**：throw `AgentNotFoundError`
- **TS-3.6 resume + SDK 不识别**（场景 X 直接通过 resume API）→ markFailed
- **TS-3.7 list filter**：layer=worker / role=DEV / status=idle 各场景过滤正确
- **TS-3.8 get 不存在**：返回 `null`（不抛）
- **TS-3.9 updateRuntimeBinding** local→cloud：record 字段更新，**不触发 resume**
- **TS-3.10 markFailed**：record.status=error + runtime_failure 字段写入

##### §3.4 SessionManager / SessionStore / TranscriptWriter（Phase B 范围 — QA 提前规划）

按 [`crash-recovery.md` 决策 4](../../packages/codeflow-runtime/docs/crash-recovery.md) 的"元数据 JSON + 事件流 markdown 拆分"模式：

- **TS-4.1** SessionStore 单 record per file：每个 session 落 `.codeflow/state/sessions/<id>.json`
- **TS-4.2** TranscriptWriter append-only：每个 RuntimeEvent 追加到对应 transcript markdown，不 atomic-rename（因为 append-only 不需要）
- **TS-4.3** 高频事件流（如 1000 事件/s）→ append 不应丢事件（用 line-based 校验 transcript 行数）
- **TS-4.4** Session 元数据更新（status=running→completed）→ atomic-write SessionStore（按决策 1 同款模式）
- **TS-4.5** 跨 run 累计 cost 字段（`total_cost_usd`）正确累加
- **TS-4.6** 启动期 reconciliation 扫描 status=running 的 session → 决定恢复策略（继续 stream / 标 cancelled / 标 failed —— 实际策略由 Phase B 实施期间敲定，**QA 在文档里列出这个 TBD**）

##### §3.5 Task Scheduler chokidar inbox（Phase C 范围）

- **TS-5.1** chokidar watch 触发：往 inbox 投一个 TASK-*.md → scheduler 读 + dispatch
- **TS-5.2** Task front-matter 不合 §3.3 Task Schema → reject + 写 reject 原因到 task 同名 .reject.md（schema 设计待定，QA 在文档里提议）
- **TS-5.3** state_history 自动追加：每次 status 变化 → state_history 数组追加一条（不删旧的）
- **TS-5.4** 大量 task 进入 inbox（如 50 个）→ scheduler 不丢失，按 priority 排序

#### §4 测试基础设施（QA 给 DEV 的建议）

- 测试框架推荐：`node:test`（Node 20+ 内置；零依赖；与 v0.1 "轻量级" 哲学契合）
- 备选：`vitest`（如 DEV spike 已选）
- **不推荐**：jest / mocha（过重；且 jest 的 ESM 兼容仍有坑）
- mock 策略：避免 `jest.mock` 类型 monkey-patch；推荐用 InMemorySdkAdapter 显式注入
- 覆盖率门槛建议：行覆盖率 ≥ 80%、分支覆盖率 ≥ 70%（Phase A 末由 DEV 报告）
- CI 集成：本任务 *不* 配置 CI，留待 Sprint S6（E2E 验收）阶段统一接
