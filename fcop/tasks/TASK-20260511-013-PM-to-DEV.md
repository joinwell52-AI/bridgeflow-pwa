## §一 PM 第 15 次错误自披露（派单前自我拦截）

### 错误内容

PM TASK-20260511-007 §四 Day 4.1 写：

> 「Day 4.1：`AgentRegistry` 改造 — 让 agent listing 走 fcop `Project.list_agents$()`」

**真实情况**：fcop@1.1.0 的 `Project` 类**根本没有 `list_agents` 方法**。本地 `inspect.getmembers()` 实证：

```
fcop.Project 公开方法（带 agent / list / role 关键词的全部）：
  deploy_role_templates  — 部署角色模板文件
  list_issues            — list issues
  list_reports           — list reports
  list_reviews           — list reviews
  list_tasks             — list tasks
  role_occupancy         — 角色占用情况（基于 task 推算）
```

**没有 `list_agents`** —— 因为 **fcop 不管 agent registration**（fcop 是「文件协作协议」，只管 tasks/reviews/reports/issues 4 种文件 + role_occupancy 视图，没有「agent」实体概念）。

### 错位类型

第 9 条自约束 path 版三件套**首次正面拦截**（vs 第 11/12 次拦截在 DEV 实施时才发现）：

| 三件套维度 | 结果 |
|---|---|
| 1. path 存在 | ✅ AgentRegistry / InboxWatcher 文件均在 `packages/codeflow-runtime/src/registry/` 和 `scheduler/` |
| 2. public method 暴露 | ❌ **fcop@1.1.0 Project 不暴露 `list_agents`** |
| 3. 参数类型匹配 | 不适用（方法不存在） |

错位本质：**PM 想当然把 fcop 当成「agent 管理库」** — 实际 fcop 只管文件 + role-based 占用视图。CodeFlow 的 AgentRegistry（agent 元数据 + Cursor SDK agent_id 对账）**与 fcop 的 role_occupancy 不重叠**，没必要去 routing。

### 处置

**撤回 Day 4.1**（AgentRegistry → fcop）。Day 4 范围**收窄到 InboxWatcher inspectTask wire**（Day 3 推迟过来的）+ DEV 自决可选小项。

### 给 PM 的 emergence-log 启示

## §二 Day 4 范围（修正后）

### §2.1 主交付 — InboxWatcher inspectTask wire

**目标**：让 `InboxWatcher` 在新 TASK 文件落 inbox 时，**先用 `fcopClient.inspectTask(filename)` 跑 fcop schema 校验**，再决定是否往下传给 `TaskDispatcher`。

**路径**：`packages/codeflow-runtime/src/scheduler/InboxWatcher.ts`（PM 已 ripgrep 核对）

**接口（参考，DEV 可优化）**：
- `InboxWatcherOpts` 加 `fcopClient?: FcopProjectClient`（与 Day 2 TaskParser / Day 3 ReviewWriter 同样 optional 注入模式）
- `InboxWatcherOpts` 加 `onValidationFail?: 'reject' | 'needs_human_review' | 'dispatch_anyway'`（DEV 自决默认值；PM 推荐 `dispatch_anyway` 兼容 v0.1 行为 + log warning，让 Day 5/6 再决定升级）

**fcop API 实证**（PM 已 inspect.signature 核对）：
```python
Project.inspect_task(filename_or_id: str) -> list[ValidationIssue]
ValidationIssue 字段：severity ∈ {'error', 'warning', 'info'} / field / message / path
```

**期望行为**：
1. 文件落 `docs/agents/tasks/TASK-*.md` → InboxWatcher 触发
2. 如果 `fcopClient` 已注入 → `client.inspectTask(filename)` 拿 issues
3. 如果 `issues.length === 0` → 与 Day 2 路径相同，dispatch
4. 如果有 `severity === 'error'` → 走 `onValidationFail` 路径（v0.3 默认 dispatch_anyway + log warning，与 Day 1 TaskParser yaml fallback 容错精神一致）
5. 如果 `fcopClient` 未注入 → 跳过 inspect，行为与 Day 1 完全一致（back-compat）

### §2.2 测试覆盖（DEV 推荐 3 条新 test）

| Test | 验证 |
|---|---|
| `TS-IW-D4-1` | InboxWatcher without fcopClient → 完全等同 Day 1 行为（back-compat）|
| `TS-IW-D4-2` | InboxWatcher + fcopClient stub + issues=[] → dispatch（happy path）|
| `TS-IW-D4-3` | InboxWatcher + fcopClient stub + severity='error' issue → onValidationFail 路径触发（DEV 选定默认值后钉死契约）|

可加 `TS-IW-D4-4` 覆盖 `FcopClientError` 降级路径（DEV 自决，参照 Day 2 TS-TP-D2-3）。

### §2.3 Runtime 注入

`Runtime.start()` 把 `fcopClient` 也传给 `InboxWatcher`（与 Day 2 TaskParser / Day 3 ReviewWriter 一致）。
