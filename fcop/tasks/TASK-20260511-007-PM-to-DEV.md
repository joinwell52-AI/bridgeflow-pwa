## 三、范围矩阵（删 / 改 / 保 / 不动）

### 3.1 删除（v0.1 自有实现，被 fcop 全覆盖）

| 文件 / 模块 | 删除范围 | 替代 |
|---|---|---|
| `packages/codeflow-runtime/src/_internal/task-frontmatter.ts`（如存在）| 整文件 | fcop `Project.write_task$()` 自动生成 |
| `TaskDispatcher` 内手写 frontmatter 段 | 调 fcop API 处的旧代码 | `fcopProject.write_task$({ sender, recipient, ... })` |
| `ReviewEngine.writeReview()` 内手写 frontmatter 段 | 同上 | `fcopProject.write_review$({ reviewer_role, subject_type, ... })` |
| `NeedsHumanGate` 内 console.log 升级 + 自有 audit | 升级路径 | `fcopProject.mark_human_approved$({ approver, decision, ... })` |
| `task.schema.json` / `review.schema.json`（TS types）| 删除 (CodeFlow 自己的 type 定义) | 从 fcop@1.1.0 `fcop://schemas/...` 推导（运行时读，或 codegen 时间生成）|

### 3.2 改造（v0.1 仍有价值，但调用点改走 fcop）

| 模块 | 改造 |
|---|---|
| `TaskDispatcher.dispatchOne()` | 解析任务 → 走 fcop `read_task$()` 拿对象（**不**自己 parse frontmatter）；archive 走 fcop `archive_task$()` |
| `InboxWatcher` | 仍负责文件系统 push（fcop 是拉模型，CodeFlow 加 push 优化）；watch 到新文件后调 `list_tasks$()` 让 fcop 校验 schema |
| `AgentRegistry.list()` | 团队/角色查询走 fcop `get_team_status$()` + `init_project$()`；agent 实例运行态（running/idle）仍在 codeflow-runtime 内部（Charter 5.4 涌现：fcop 没有运行态 schema，这是 CodeFlow 应用层职责，**不向 fcop 提**）|

### 3.3 保留并强化（CodeFlow 独有差异化）

| 模块 | 保留理由 |
|---|---|
| `AgentSdkAdapter`（@cursor/sdk 驱动） | fcop 不驱动 LLM/SDK |
| `RuntimeBootstrap`（crash recovery + atomic-write retry） | fcop Project API 无 Windows EPERM retry |
| `SessionManager`（agent-task SDK session 状态） | fcop 无 SDK session 概念 |
| `ReviewEngine.extractText()` + verdict 解析 | SDK message parsing 是应用层 |
| runtime `agents.json` 持久化 | fcop fcop.json 是团队级，CodeFlow 需要实例级 |
| 109+ 测试基线（保留可保留的） | 减但不删完 |

### 3.4 不动（明令）

- ❌ 不动 `AgentSdkAdapter.ts` 任何逻辑（resume-path model 仍传 / `local: { force: true }` 仍 / extractText probe 4 仍）
- ❌ 不动 `SessionManager`
- ❌ 不动 `_internal/atomic-write.ts`
- ❌ 不动 SkillRegistry / MCPInjector stub
- ❌ 不动 codeflow-shell main.ts banner（仅 §六新增 PYTHON_BIN 检查段）
- ❌ 不动 v0.2.0-beta.3 已 closed 的 7 BUG（SDK-001~007）回归保护
