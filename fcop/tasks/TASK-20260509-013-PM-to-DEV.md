## 主交付 2：`SessionStore` 实现（`packages/codeflow-runtime/src/session/SessionStore.ts`，新文件）

按 [`crash-recovery.md` 决策 4](../../packages/codeflow-runtime/docs/crash-recovery.md) "元数据 JSON + 事件流 markdown 拆分" 模式：

```ts
class SessionStore {
  constructor(opts: { dir: string });   // 默认 .codeflow/state/sessions/

  async save(record: SessionRecord): Promise<void>;       // 单 record per file: <dir>/<session_id>.json
  async load(sessionId: string): Promise<SessionRecord | null>;
  async listAll(): Promise<SessionRecord[]>;              // 扫 <dir>/*.json
  async remove(sessionId: string): Promise<void>;         // emergency cleanup; Phase B 仅暴露不必使用
}
```

实现要点：

- 单 record per file（区别于 PersistentStore 的"全 records 一个 file"）—— 因为 sessions 数量级 >> agents，per-file 避免单文件膨胀 + 并发写竞争
- `save` 用同款 atomic-write（`writeFile(*.tmp)` → `rename` → 父目录 fsync 跨平台守护）—— 复用 `JsonFileStore` 已有逻辑或抽取共用 `atomicWriteJson(path, data)` helper
- `load` 文件不存在 = 返回 `null`（**不抛**——区别于 PersistentStore.loadAll，因为这里是单 record，"找不到"是合法状态）
- `load` 解析失败 = throw `RegistryWriteError`（重用 Phase A 错误类）
- `listAll` 扫描目录，跳过 `.tmp` 后缀 + 跳过解析失败的（log warn 但不阻断）—— 这是 reconciliation 阶段需要的"宽容读"

**注意**：`crash-recovery.md` 决策 4 末尾提到"启动时 reconciliation：扫描 .codeflow/state/sessions/ 下 status=running 的 SessionRecord，调用 SessionManager 决定如何恢复（继续 stream / 标 cancelled / 标 failed）"——但这个 *逻辑* 留给 Phase C / S4 settle，**Phase B 只交付 SessionStore 的读写 surface**，不实现 reconciliation。

## 主交付 3：`TranscriptWriter` 类（`packages/codeflow-runtime/src/session/TranscriptWriter.ts`，新文件）

按 [`crash-recovery.md` 决策 4](../../packages/codeflow-runtime/docs/crash-recovery.md) "事件流 markdown append-only" 模式：

```ts
class TranscriptWriter {
  constructor(opts: { dir: string });   // 默认 .codeflow/state/transcripts/

  /** 订阅一个 RunHandle 的事件流；返回 Unsubscribe */
  attach(runId: string, handle: RunHandle): Unsubscribe;

  /** 主动追加一行（如 cancelSession 的 reason） */
  append(runId: string, kind: TranscriptEntryKind, text: string): Promise<void>;

  /** 关闭 + flush（finalize on session.ended_at） */
  close(runId: string): Promise<void>;
}
```

实现要点：

- 文件路径：`<dir>/<run_id>.md`
- **append-only** —— 用 `fs.promises.appendFile` 或保持 fd 打开做高频 write
- 每行一条 entry：`[ISO timestamp] [event_type] payload_summary` 风格 markdown
- **不**做 atomic-write（因为 append-only 不需要 — 部分写入丢的就是丢的，下次启动从 SDK 恢复）
- 12 类 RuntimeEvent（8 SDK + 4 runtime）必须能订阅：
  - SDK 8 类：`message_start / message_delta / message_end / tool_call_start / tool_call_end / thinking_start / thinking_end / error`（具体类型对照 spike `_ignore/spike_sdk_doorbell/sender.ts` 已验证）
  - runtime 4 类：`runtime_session_started / runtime_session_ended / runtime_session_cancelled / runtime_persistence_flushed`
- `close()` 写一行 `[ISO] [session_ended] reason: ...`，然后断开 fd（如果保持的话）

**性能要求**：
- TS-4.3（QA 测策场景）= 1000 事件/s 不丢失。建议实现：保持 fd 打开 + 批量 buffer 但 1s 内强制 flush；不要每事件都打开-关闭。
- 如要避免 `appendFile` 在 Node 老版本的潜在问题，可用 `fs.createWriteStream(path, { flags: "a" })` 拿一个 stream 持有

- 这个字段是**让 agent 自己决策依据**（保留），还是
