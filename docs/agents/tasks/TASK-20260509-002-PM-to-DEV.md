# Sprint S2：Agent Registry + Session Manager 设计骨架（不写实现）

## 背景

S1 已经把 §3 五类 schema + validator + CLI + 9 fixtures 落到 `packages/codeflow-protocol/`，证明协议层可执行（详见 `REPORT-20260509-001-DEV-to-PM.md`）。但 schema 只是**协议骨架**，没有把 schema 跑起来的运行时——按 §10.2 路线图，下一步是 **Sprint S2**：让 Runtime 有能力管理 Agent 实例。

本 sprint **只做设计骨架，不写功能实现**。原因：
- S2 的接口设计需要在和 §3 schema 接口对齐 + §2.1 子系统总览的精确映射后才稳；如果接口不稳就写实现，等到 S3/S4 反向倒逼接口改动 = S2 白做。
- S2 的实现细节（持久化策略、crash recovery、并发模型）需要 ADMIN 在 review 阶段拍板技术决策（如：是否用 SQLite / 仅 JSON / per-agent 单 inbox 文件夹等）；先把骨架交出来供 review，比直接写代码更省周期。
- 这是 §0.8.2 第 4 条硬约束「每步必须有 reviewer」在 sprint 内部的兑现：S2 设计 → S3/S4 实现，中间过 ADMIN 一道。

### 必交付 2：Agent Registry 接口设计（`AgentRegistry.ts`）

按 §2.1 子系统 3「PCB 表」+ §3.2 Agent Schema 的映射，给出**完整 TypeScript 接口签名 + JSDoc**（方法体 throw not-implemented），至少含：

| 方法 | 用途 | 关键不变量 |
|---|---|---|
| `register(agentSpec: AgentSchema): Promise<AgentRecord>` | 首次 `Agent.create()` 包装 + 写入 agents.json | layer=admin 必须拒绝（详见 §3.2 layer 字段约束） |
| `resume(agentId: string): Promise<AgentRecord>` | 进程崩溃后从 agents.json 重建 | resume 失败必须给可读错误 + 不破坏 agents.json |
| `list(filter?: { layer?, role?, status? }): Promise<AgentRecord[]>` | 列出 agent | filter 默认空 = 全部 |
| `get(agentId: string): Promise<AgentRecord \| null>` | 单条查询 | 不存在 = null，不抛异常 |
| `updateRuntimeBinding(agentId, runtime: 'local' \| 'cloud'): Promise<void>` | 切换执行节点 | 触发自动重新 resume |
| `markFailed(agentId, error: string): Promise<void>` | 标记 agent 进入 failed 态 | failed agent 不能再 send，必须先 reset |

**接口设计原则**（PM 在派单时已与 ADMIN §0.6.4 「kernel 四件套」对齐）：
- 所有方法 async，因为 SDK `Agent.create/resume` 都是 async（详见 spike 验证）
- 所有方法都对 §3.2 Agent Schema 做 *入参* 校验（调用 `codeflow-protocol` 的 validator）
- 所有写操作都必须先 ajv-validate，写入失败 = 不污染 agents.json（写临时文件 + atomic rename，详见 *S3 实现期* 才做的细节）
- **layer 字段约束在 Registry 入口处校验**：`register({layer: 'admin'})` 必须 throw，不能等到 SDK 那边才 fail

### 必交付 3：Session Manager 接口设计（`SessionManager.ts`）

按 §2.1 子系统 1「会话三元组」+ §3.5 Session Schema 的映射，给出接口签名 + JSDoc：

| 方法 | 用途 | 关键不变量 |
|---|---|---|
| `startSession(agentId, taskId, payload): Promise<SessionHandle>` | 开始一个 agent×task×run 的会话 | session.status = 'running'；同 agent 多 task 必须串行（除非 §3.2 explicit concurrency 字段允许并行） |
| `getSession(sessionId): Promise<SessionRecord \| null>` | 查询会话 | 不存在 = null |
| `listActive(): Promise<SessionRecord[]>` | 列出所有 running session | 用于 §10.3 Mobile Console "Agent 状态" 屏 |
| `cancelSession(sessionId, reason): Promise<void>` | 优雅取消 | 必须调用 SDK `run.cancel()` + 落 SessionRecord.status='cancelled' + 写入 cancel-reason 到 transcript |
| `cancelAllForEmergencyStop(): Promise<{cancelled: string[]}>` | §0.9.5 红按钮：取消所有 running session | 必须落盘 EMERGENCY-{ts}.md（schema 详见 v0.2 sprint S10，S2 只留接口） |
| `onEvent(handler: (event: RuntimeEvent) => void): Unsubscribe` | 订阅 SDK 流事件（用于 transcript / Mobile push） | 所有 8 类 SDK 事件都必须能订阅到（详见 spike `sender.ts` 已验证） |

### 必交付 4：State 类型定义（`types/state.ts`）

`AgentRecord` / `SessionRecord` / `RunHandle` / `RuntimeEvent` 等核心类型 ——**直接基于 `@codeflow/protocol` 的 schema 类型派生**，不允许自己重新发明字段名（这是 §8.0 硬规则 #4 在代码层的兑现）。

如果 schema 类型不够用（譬如 SessionRecord 需要 `lastEventAt` 字段而 §3.5 Session Schema 没有），**必须在本任务回执里列出**，不允许悄悄在 runtime 包里加。后续按 §3.3.1.b 唯一合法升级路径走 `D:\FCoP` 评审。

### 必交付 5：crash-recovery 设计文档

新增 `packages/codeflow-runtime/docs/crash-recovery.md`，回答 4 个问题（仅文档，不实现）：

1. **agents.json 何时写入？** — 选项：每次 Registry 写操作后立刻 / 定期 batch / 仅退出时落盘 → 推荐方案 + 理由

文档长度 ≤ 200 行，结论必须明确，不接受「待 S3 决定」。
