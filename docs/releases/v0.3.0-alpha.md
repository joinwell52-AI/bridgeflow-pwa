## TL;DR

CodeFlow v0.3 makes `fcop@1.1.0` the **first-class runtime backend** for the task / review / human-approval file layers, accessed in-process via `pythonia`. The v0.1 YAML emitter path is preserved as a graceful fallback for any layer where the fcop bridge is degraded (`PYTHON_BIN` unset, `fcop` import fails, etc.) вҖ” sameйҷҚзә§е®№й”ҷ spirit as Day 1 `CODEFLOW_SKIP_FCOP_PROBE=1` mode.

**Charter 5 alignment**: this is the first incremental step of Charter 5 (В«CodeFlow = FCoP application; CodeFlow's own task/review writers are deprecated once fcop carries the loadВ»). Day 5 clarified the schema ownership matrix (Charter 5.4 вҖ” 4 schemas remain CodeFlow-owned application-layer emergence; only `Review` is a dual contract with fcop). Day 6 finalized the version bump + release notes (this file).

**Compatibility**: drop-in for `v0.2.0-beta.3` consumers вҖ” `RuntimeCreateOptions.fcopClient` is optional everywhere; `null` keeps every subsystem on v0.1 behavior. No v0.2 caller signatures break.

## What's new

### 1. `pythonia` bridge + `FcopProjectClient` adapter (Day 1)

New module `packages/codeflow-runtime/src/_external/fcop-client.ts` (~1100 lines) exposes a thin, type-safe TS faГ§ade over `fcop@1.1.0` accessed through `pythonia`:

- `FcopProjectClient.create({ projectRoot, workspaceDir, ensureInitialized, team, lang })` вҖ” factory with workspace-layout escape hatch (`workspaceDir: "fcop"` preserves CodeFlow v0.x layout per DEV-005 В§S8).
- `assertFcopReady()` вҖ” boot-time probe returning `{ fcopVersion, pythonVersion, pythonExecutable }`; throws `FcopClientError` with PYTHON_BIN / `Python<3.10` / `fcop@1.1.0` actionable hints.
- `disposeFcopBridge()` вҖ” graceful shutdown of the pythonia child Python process.
- Lazy `pythonia` import (the npm module spawns Python synchronously on import вҖ” we defer it via `await import("pythonia")` so probe can `try/catch` ENOENT).
- 12 spike-discovered surprises (S2-S12 from `TASK-20260511-005`) are pre-paid: kwarg-syntax (`fn$(kwargs)` not `fn(key$, ...)`), enum-via-`.value`, `inspect.signature` regression every change.

### 2. `TaskParser` instance API (Day 2)

`TaskParser` is now a class with both a static and instance form:

- `TaskParser.parse(filepath)` вҖ” unchanged; YAML path; preserves the 4 baseline `TS-5.4..5.6` tests.
- `new TaskParser({ fcopClient }).parse(filepath)` вҖ” fcop-first: delegates to `fcopClient.readTask(basename(filepath))`; on `FcopClientError`, automatically falls back to the YAML disk parser so a degraded fcop bridge never breaks dispatch ("Path A ж”№иүҜ" model).
- `Runtime.create()` wires the instance form whenever `opts.fcopClient` is supplied; otherwise stays on the static form.

### 3. `FcopTask` interface aligned to `fcop@1.1.0` actual shape (Day 2 fix)

Day 1 shipped a flat `FcopTask` that did not match fcop's actual `Task` dataclass (which uses a **nested** `frontmatter: TaskFrontmatter`). This was a latent crash bug вҖ” flat reads of `task.sender` etc. would return `undefined` from the real fcop bridge.

Day 2 fix:

- `FcopTask` is now top-level: `task_id / filename / path / date / sequence / body / is_archived / frontmatter`
- `FcopTaskFrontmatter` is a separate interface: `protocol / version / sender / recipient / priority / thread_key / subject / references / risk_level / extra`
- Convenience top-level accessors (`task.sender`, `task.priority`, etc.) are pre-pulled from `frontmatter` so callers don't need to descend.
- New `readTask(filenameOrId)` public method; PM TASK-009 В§еӣӣ assumed this existed (PM error #11) and Day 2 added it.

### 4. `ReviewWriter` instance fcop API (Day 3)

- `new ReviewWriter({ reviewsDir, fcopClient })`: same "Path A ж”№иүҜ" model as `TaskParser`.
- fcop-first path discards `verdict.review_id` / `verdict.human_approval` / `verdict.decision_duration_ms` (fcop owns id generation, schema rejects v0.1 stub ack block, and decision_duration is not part of fcop schema).
- `FcopClientError` вҶ’ YAML fallback to `<reviewsDir>/<review_id>.md`.
- When both fcop AND fallback yaml fail вҶ’ `ReviewWriteError` with the original fcop cause preserved.

### 5. `NeedsHumanGate.markApproved()` new public method (Day 3)

- `gate.markApproved(reviewId, spec)` вҖ” called by upstream (v0.2 Mobile / v0.5 ack queue) after a real human ack arrives.
- If `fcopClient` is wired вҶ’ forwards to `fcopClient.markHumanApproved(reviewId, spec)`; the returned `HumanApproval` reflects fcop's actual `approved_at` (not our wall-clock guess).
- If `fcopClient` is null вҶ’ degraded in-memory mode: returns a fully-populated `HumanApproval` but does NOT touch any file. v0.3 contract: "no fcop = no persisted audit."
- v0.1 `push()` semantics are unchanged.

### 6. `FcopReview` interface aligned to `fcop@1.1.0` actual shape (Day 3 fix)

Day 1 shipped a `FcopReview` that omitted `body / date / mtime` and didn't include `human_approval.approved_at / evidence`. Unlike `FcopTask` (which crashed because Task is nested), `Review` is fully top-level so the partial read worked вҖ” just gave callers fewer fields.

Day 3 fix:

- `FcopReview` adds `body / date / mtime` (top-level).
- `FcopHumanApproval` is now a named interface: `approver / decision / approved_at / channel / comment / evidence`.
- `evidence` is a `Record<string, unknown> | null` (Mobile/CLI auth-method, device_id, ip вҖ” surfaced as a plain object; v0.5 narrowing will type it further).
- New `readReview(filenameOrId)` public method (PM error #14 вҖ” Day 1 didn't expose it; Day 3 ships it).

### 7. `FcopValidationIssue` + `inspectTask()` (Day 3 / 4)

- `FcopValidationIssue`: `severity ('error'|'warning'|'info') / field / message / path (string | null)`.
- `FcopProjectClient.inspectTask(filenameOrId)` вҖ” runs fcop's schema validator; returns `FcopValidationIssue[]`.
- Day 3 ships the public method; Day 4 wires it into `InboxWatcher`.

### 8. `InboxWatcher` fcop pre-dispatch schema gating (Day 4)

- `new InboxWatcher({ dir, fcopClient, onValidationFail })`: optional fcop bridge for boot-time schema gating.
- Default `onValidationFail = "dispatch_anyway"`: log warning + still dispatch вҖ” sameе®№й”ҷ spirit as Day 2 TaskParser fallback. Rationale: v0.3 schema strictness may flag some legacy v0.2 task files; we'd rather warn loudly than silently swallow operator drops while migrating.
- Other policies: `"reject"` (drop event), `"needs_human_review"` (log with explicit prefix for the v0.5 ack queue to pick up вҖ” for v0.3 it behaves like `reject`).
- `FcopClientError` from `inspectTask` always degrades to dispatch_anyway with a warning вҖ” a degraded fcop bridge MUST NEVER eat tasks.

### 9. Transparency banner additions (Day 2 / 3 / 4)

`codeflow-shell` banner now prints per-subsystem fcop wire-up state so operators see which layers route through fcop:

```
fcop bridge    : fcop 1.1.0 via pythonia (Python at ...\python.exe)
Task parser    : TaskParser=fcop                                                    вҶҗ Day 2
Review writer  : ReviewWriter=fcop + NeedsHumanGate fcop audit wired               вҶҗ Day 3
Inbox watcher  : InboxWatcher=fcop schema-gating (onValidationFail=dispatch_anyway) вҶҗ Day 4
```

When fcop is skipped or fails to construct, the same three lines fall back to `yaml fallback` / `Day-1 pass-through` strings so operators see the degradation explicitly.

### 10. Schema ownership clarification (Day 5)

v0.3.0-alpha clarifies the relationship between CodeFlow's 5 JSON schemas and fcop@1.1.0's 8 schemas. **No schema files were modified** вҖ” the change is purely documentation + comments. The framing matters because prior PM plans repeatedly assumed CodeFlow schemas are "mirrors of fcop SSOT" that must drift-verify against `D:\FCoP`; the actual relationship is more nuanced:

- **CodeFlow-owned (4)** вҖ” Agent / Task / Session / Skill: Charter 5.4 application-layer emergence schemas. fcop@1.1.0 either has no equivalent (Task lives inside `ipc-envelope` with `type=TASK`; Session is explicitly excluded per fcop ADR-0020), or describes a different facet (fcop `agent` is role identity + capability in `fcop.json`; CodeFlow `Agent` is runtime PCB in `agents.json`; fcop `skill` is a role-capability reference; CodeFlow `Skill` is an MCP server registry). These four schemas are CodeFlow's SSOT and stay in `packages/codeflow-protocol/schemas/`.
- **Dual contract (1)** вҖ” Review: fcop-first path вҶ’ fcop@1.1.0 `review.schema.json` is SSOT (via `Project.write_review$()`; v0.3.0-alpha В§S1-S3 enumerates field-by-field differences); yaml fallback path вҶ’ CodeFlow v0.1 `review.schema.json` is SSOT. Both are kept вҖ” neither is a "drift bug".
- **fcop-only (5)** вҖ” boundary / encoding / event / failure / ipc-envelope: fcop governance / transport schemas. CodeFlow does not mirror, import, or wire any of these (per fcop ADR + Charter 5.1 "CodeFlow ж°ёдёҚйҮҚеҸ‘жҳҺ FCoP е·Іе®һзҺ°зҡ„иғҪеҠӣ").

`packages/codeflow-protocol/src/types.ts` header was rewritten in Day 5 to include the **Schema Ownership Matrix** as the file's contract; each per-schema section (В§3.2-В§3.6) carries a `Charter 5.4: ...` annotation. Full ownership analysis lives in [`fcop/internal/p4-day5-schema-drift.md`](../../fcop/internal/p4-day5-schema-drift.md).

**For consumers**: when extending Agent / Task / Session / Skill, edit only `packages/codeflow-protocol/schemas/` + `src/types.ts`. When extending Review, edit BOTH CodeFlow's schema (yaml fallback) AND raise the topic upstream with `D:\FCoP` for the fcop path. Never silently align the two вҖ” that's how schema-drift bugs masquerade as harmless cleanups.

## Semantic changes operators MUST know about

### S1 вҖ” v0.3 fcop path discards `verdict.review_id` (Day 3)

On the fcop path, `ReviewWriter.write(verdict, body)` returns the **fcop-generated** filepath, not `<reviewsDir>/<verdict.review_id>.md`. Caller-supplied `review_id` is overridden by fcop's sequence-generator. Yaml fallback path preserves the v0.1 caller-controlled review_id contract.

**Impact**: `ReviewEngine` and other CodeFlow consumers should treat the returned filepath as authoritative when fcop is wired.

### S2 вҖ” v0.3 fcop path discards `verdict.human_approval` stub block (Day 3)

The v0.1 `needs_human` verdict eagerly wrote `human_approval: {pushed_to, pushed_at, approved_by: null, approved_at: null}` to the REVIEW file. fcop schema rejects this вҖ” fcop only allows `human_approval` to be filled after a real `mark_human_approved$()` call.

In v0.3 fcop path:
- The REVIEW file is written without the v0.1 stub block.
- After a real human ack lands (v0.2 Mobile / v0.5 ack queue), `NeedsHumanGate.markApproved()` calls `fcopClient.markHumanApproved()` which patches the file's front-matter with the full `HumanApproval` block (including `approver / approved_at / channel / comment / evidence`).

Yaml fallback path preserves v0.1 stub behavior.

**Impact**: audit-trail readers / Mobile must be aware that a v0.3 fcop-path REVIEW with `decision: "needs_human"` may have NO `human_approval` block at all if no human has acked yet.

### S3 вҖ” `ReviewVerdict.decision_duration_ms` not persisted on fcop path (Day 3)

fcop schema has no field for this; the fcop path drops it. Yaml fallback preserves it.

### S4 вҖ” `InboxWatcher` schema gating may add ~200-500ms per task dispatch (Day 4)

The pre-dispatch `inspectTask()` call is async + IPC; expect a small latency increase before `task_added` fires. The watcher's `add` event listener is non-blocking (chokidar keeps firing), so this does NOT slow ingestion throughput вҖ” it only delays the per-task dispatch decision.

**Operators**: this is intentional. Use `CODEFLOW_SKIP_FCOP_PROBE=1` to fall back to Day-1 zero-latency dispatch if needed.

## Compatibility & migration

### From `v0.2.0-beta.3`

вң… Drop-in. Every fcop wire-up in v0.3 is gated by `opts.fcopClient` being non-null in `Runtime.create({...})`. Existing callers that don't supply `fcopClient` see exactly v0.2 behavior.

The new `Runtime.create()` opt-in:

```ts
const fcopClient = await FcopProjectClient.create({
  projectRoot: process.cwd(),
  workspaceDir: "fcop",
  ensureInitialized: false,
});
const runtime = await Runtime.create({
  /* ... existing v0.2 opts ... */
  fcopClient,           // вҶҗ new вҖ” opt-in
});
```

When `fcopClient` is supplied, `TaskParser` / `ReviewWriter` / `NeedsHumanGate` / `InboxWatcher` automatically route through fcop. When omitted, all four subsystems remain on v0.1 YAML behavior.

### Environment variables (v0.3 additions)

| Var | Effect |
|---|---|
| `PYTHON_BIN` | Path to Python 3.10+ with `fcop@1.1.0` installed. Required for fcop bridge boot. |
| `CODEFLOW_SKIP_FCOP_PROBE` | When `"1"`, skips the boot-time fcop probe; runtime stays on Day-1 YAML behavior for all four subsystems. Used for dev environments without Python / fcop. |

The shell's `.env.example` documents both with actionable hints.
